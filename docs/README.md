## Documentation Guidelines

### When to Add New Documentation

1. **Root Level** - Only for essential, frequently-accessed documents:
   - Project setup and configuration
   - Critical guidelines for all contributors
   - High-level architecture and formulas

2. **docs/** - Active, current documentation:
   - Feature implementation guides
   - Developer references and quick starts
   - Testing and quality assurance
   - Performance and optimization

3. **docs/archive/** - Historical or completed documentation:
   - Completed initiatives and sprints
   - Superseded guides and reports
   - Historical decision records

### Naming Conventions

- Use `UPPERCASE_WITH_UNDERSCORES.md` for documentation files
- Use descriptive names: `FEATURE_IMPLEMENTATION_GUIDE.md` not `GUIDE.md`
- Add suffixes for document types:
  - `_GUIDE.md` - Comprehensive guides
  - `_QUICK_START.md` - Quick start tutorials
  - `_QUICK_REFERENCE.md` - Reference cards
  - `_SUMMARY.md` - Executive summaries
  - `_REPORT.md` - Analysis reports

### Keeping Documentation Current

- Archive completed initiative documentation within 2 weeks of completion
- Update this index when adding new documentation
- Remove obsolete documentation rather than letting it languish
- Link related documents together
